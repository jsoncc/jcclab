# 本站总访问量（PV）与总访客（UV）实现说明

> **一句话**：静态站自己不能存计数；用 **Cloudflare Worker** 提供 `GET /stats`，用 **KV** 持久化两个数字；**UV** 靠 **Cookie** 区分是否算新访客；页脚 **`fetch` + credentials** 拉数据并展示。

---

## 1. 架构速览

| 层级 | 做什么 |
|------|--------|
| **浏览器 / GitHub Pages** | 打开站点 → `App.vue` 在 `onMounted` 里请求统计接口（带 Cookie） |
| **Cloudflare Worker** | 同域名根路径还负责百度翻译 **POST**；**`GET /stats`** 单独走统计逻辑 |
| **Cloudflare KV** | 存 `total_pv`、`total_uv` 两个字符串键，每次请求读后写回 |
| **本地开发** | Vite 把 **`/site-stats`** 代理到本机 **`127.0.0.1:8787/stats`**（需另起 `wrangler dev`） |

数据流（生产）可理解为：

`用户打开 Pages` → `fetch(Worker根URL/stats, { credentials: 'include' })` → `Worker 改 KV` → `返回 JSON` → `页脚更新文案`。

## 3. 接口约定：`GET /stats`

### 3.1 成功且已配置 KV

- **方法**：`GET`
- **响应 JSON**（示例字段）：
  - `totalPv`：总访问量（每次请求 **+1**）
  - `totalUv`：总访客（**无**指定 Cookie 时本次 **+1**）
  - `configured`：`true`

### 3.2 未绑定 KV（`env.SITE_STATS` 不存在）

- `configured: false`，`totalPv` / `totalUv` 为 0，并带 `message` 提示查 `workers/README.md`。

### 3.3 来源不在白名单

- HTTP **403**，JSON 里 `error: 'origin_not_allowed'`。

### 3.4 KV 键名（固定）

| KV 键 | 含义 |
|-------|------|
| `total_pv` | 总访问量 |
| `total_uv` | 总访客 |

值为十进制整数字符串；不存在时按 0 处理。

### 3.5 访客 Cookie

| 项 | 说明 |
|----|------|
| 名称 | `jsoncc_stats_visitor` |
| 值 | `1` |
| 作用 | 请求里**已有**该 Cookie → 只加 **PV**，**不加 UV**；否则 **UV+1** 并 `Set-Cookie` |
| 有效期 | 约 1 年（`Max-Age`） |
| 本地 http | `SameSite=Lax`，**无** `Secure`（否则本地浏览器会丢 Cookie） |
| 生产 https | `SameSite=None; Secure`（跨站带 Cookie 到 Worker 域名所需） |

## 4. 为什么 CORS 要配白名单？

页脚请求使用 **`fetch(..., { credentials: 'include' })`**，浏览器要求：

- 不能 `Access-Control-Allow-Origin: *`
- 必须返回**具体**的 `Access-Control-Allow-Origin`（与请求 `Origin` 一致）
- 并带 `Access-Control-Allow-Credentials: true`

代码里 **`workers/stats.ts`** 维护 **`DEFAULT_ORIGINS`**（含 `https://jsoncc.github.io`、`http://localhost:3000` 等）。若站点改用**自定义域名**，需在 Worker 配置 **`STATS_ORIGINS`**（`wrangler.toml` 里 `[vars]`，逗号分隔）或在源码白名单中追加。

**补充**：部分场景下浏览器不发 **`Origin`**（例如地址栏直接打开经代理的 URL），实现里会用 **`Referer`** 在白名单内推断来源（仍仅限已允许的协议+主机）。

## 5. 前端如何决定请求地址？

逻辑在 **`src/App.vue`** 的 **`buildStatsEndpoint()`**（优先级从上到下）：

1. 若配置了 **`VITE_SITE_STATS_URL`**：用该地址；若以 `/` 结尾会去掉；若路径未以 **`/stats`** 结尾会自动拼上 **`/stats`**。
2. 否则若有 **`VITE_BAIDU_TRANSLATE_URL`**：**Worker 根地址 + `/stats`**（与翻译共用同一个 Worker）。
3. 否则若在 **开发模式**：返回 **`/site-stats`**（交给 Vite 代理到本机 Wrangler）。
4. 否则：**`null`**，页脚提示未配置。

请求统一 **`credentials: 'include'`**，以便携带/接收 Worker 域名下的访客 Cookie。

## 6. 生产环境（GitHub Pages）要配什么？

- **Actions Secrets**（与构建时注入一致）：
  - **`VITE_BAIDU_TRANSLATE_URL`**：已部署的 Worker **根 URL**（无末尾 `/` 即可），统计即 **`该URL/stats`**。
  - 可选 **`VITE_SITE_STATS_URL`**：统计与翻译**不同 Worker** 时，填完整统计 URL（通常以 **`/stats`** 结尾或会自动补全）。

- Worker 侧必须在 **`wrangler.toml`** 中为 **`SITE_STATS`** 绑定**真实 KV 命名空间 id**；具体创建与写入方式见 **`workers/README.md`** 或根目录 **`npm run workers:kv-bind`**。

---

## 7. 本地开发怎么联调？

1. **终端 A**：项目根执行 **`npm run workers:dev`**（默认监听 **8787**，提供 **`/stats`**）。
2. **终端 B**：**`npm run dev`**（Vite **3000**）。
3. 页脚会请求 **`http://localhost:3000/site-stats`** → 代理到 **`http://127.0.0.1:8787/stats`**。

若 **8787 未启动**，代理会失败，页脚会提示检查 Worker；Vite 对代理错误会返回说明性文案（如 **503** 与提示执行 `workers:dev`）。

本地 KV 缓存目录一般在 **`workers/.wrangler/`**（已在 **`.gitignore`**，勿提交）。

## 8. 局限与认知

| 点 | 说明 |
|----|------|
| **UV 非精确科学** | 清 Cookie、隐私模式、多设备/浏览器会重复计；第三方 Cookie 策略收紧时，Cookie 可能不稳定。 |
| **PV** | 每次进入页面并执行到 **`loadSiteStats`** 会 **+1**（含刷新）；爬虫若命中接口也会涨（未做复杂防刷）。 |
| **与百度翻译** | 同一 Worker、同一根 URL；**POST** 仍转发百度，**GET /stats** 只走统计，互不混用路径即可。 |

---

## 9. 常见问题速查

| 现象 | 可检查项 |
|------|----------|
| 页脚提示未配置 | 生产是否设置 **`VITE_BAIDU_TRANSLATE_URL`**；构建日志里是否注入成功 |
| `configured: false` | **`wrangler.toml`** 是否绑定 **`SITE_STATS`**；是否已 **`wrangler deploy`** |
| 403 / `origin_not_allowed` | 当前站点 **Origin** 是否在白名单；自定义域是否写入 **`STATS_ORIGINS`** |
| 本地一直失败 | 是否已 **`npm run workers:dev`**；**8787** 是否被占用 |
| 线上 UV 异常 | 浏览器第三方 Cookie 设置；是否跨站场景需 **`SameSite=None; Secure`**（生产已按 https 处理） |
