# Git 提交信息（Commit Message）格式规范

**业界最标准、最通用、Cursor / GitHub / 团队开发都认可的格式**

---

# 一、标准格式（必须记住）

```
<类型>: <简短描述>
```

例子：

```
feat: 添加历史上的今天卡片展示
fix: 修复图片路径404问题
docs: 更新README部署说明
```

---

# 二、最常用的 7 种类型


| 类型           | 含义       | 你的项目使用场景          |
| ------------ | -------- | ----------------- |
| **feat**     | 新增功能     | 加页面、加组件、加接口、加模块   |
| **fix**      | 修复bug    | 修复图片不显示、部署失败、样式错乱 |
| **docs**     | 文档修改     | 改README、加说明、写使用文档 |
| **style**    | 样式/格式调整  | 改CSS、调整排版、格式化代码   |
| **refactor** | 代码重构     | 优化结构、改名、抽离逻辑，不改功能 |
| **test**     | 测试相关     | 加测试代码             |
| **chore**    | 构建/依赖/杂项 | 打包配置、升级依赖、清理文件    |


---

# 三、正确示例（你可以直接复制）

```
feat: 新增历史上的今天详情页面
fix: 修复百度翻译部署失败问题
docs: 完善项目部署教程
style: 优化卡片列表样式间距
refactor: 重构图片目录结构
chore: 升级vite配置
```

---

# 四、错误示范（不要这样写）

```
提交代码
修复
更新
aaa
111
测试
```

❌ 不规范、不清晰、别人看不懂、自己以后也忘

---

# 五、最适合你项目的真实提交示例

```
feat: 新增阿里云OSS图片存储方案
fix: 修复GitHub Pages图片404问题
docs: 更新Tag版本发布说明
style: 优化博客文章布局
refactor: 统一图片目录结构
chore: 优化打包体积
```

---

# 六、一句话记住规范

**类型: 做了什么**
英文小写 + 冒号 + 空格 + 描述

---

# 七、如果你想更规范（可选，进阶）

```
<类型>(<范围>): <描述>

例：
feat(home): 首页添加历史卡片列表
fix(image): 修复图片加载失败
docs(readme): 更新部署步骤
```

---

## 总结（你只需要背这一句）

**类型: 简短清晰的描述**
例如：

```
fix: 修复图片路径问题
docs: 更新版本说明
feat: 新增功能模块
```
